# s3lo v1.9.0 — Image Signing & Verification Design

**Date:** 2026-04-12  
**Issues:** #20 (`s3lo sign`), #21 (`s3lo verify`)  
**Status:** Approved

---

## Goal

Add supply-chain security to s3lo by allowing images to be cryptographically signed and
verified. Signatures prove that a specific digest was produced by an authorized signer.
This satisfies SOC2 and FedRAMP audit requirements for artifact integrity and non-repudiation.

---

## What Gets Signed

The **manifest digest** (`sha256:...`) of the resolved image tag. The digest is a SHA-256
hash of the manifest JSON, which in turn references all image layers by their digests.
Any change to any layer or the manifest itself invalidates the signature.

Tags are resolved to digests at sign time and stored in the signature — so verifying
`s3://bucket/myapp:v1.0` checks that the *current* manifest matches the stored signature.

---

## Key Backends

| Backend | Flag value | Use case | FIPS compliant |
|---|---|---|---|
| AWS KMS | `awskms://alias/name` or `awskms:///arn:aws:kms:...` | Production / FedRAMP | Yes (HSM, Level 2+) |
| Local key file | `./cosign.key` / `cosign.key` | Dev, on-prem, CI with secrets manager | No (key in userspace) |
| HashiCorp Vault | `hashivault://host/v1/transit/keys/name` | Multi-cloud on-prem, Vault FIPS mode | Yes (Vault Enterprise FIPS) |

### AWS KMS — compliance notes
- Private key never leaves the KMS HSM; signing happens server-side.
- Every `kms:Sign` and `kms:Verify` call is recorded in CloudTrail → automatic audit log.
- Access controlled by IAM: separate signing role from push role satisfies separation of duties.
- Key rotation built in (`EnableKeyRotation`).
- Recommended default for any deployment targeting SOC2 or FedRAMP.

### Local key file
- Standard cosign key format (ECDSA P-256).
- Private key is password-encrypted at rest (`COSIGN_PASSWORD` env or prompt).
- Acceptable for dev and CI when the key file is injected from a secrets manager (not checked in).

### HashiCorp Vault
- Uses Vault Transit engine; private key never leaves Vault.
- Vault Enterprise supports FIPS 140-2 mode.
- All operations written to Vault audit log.

### Not in scope for v1.9.0
- GCP KMS, Azure Key Vault (will follow naturally in v1.10.0 multi-cloud work).
- Keyless / Sigstore OIDC (not FIPS-compliant, not the target audience).
- PKCS11 hardware tokens (YubiKey etc.) — future enhancement.

## Signature Storage

Stored in S3 alongside the manifest, one file per signing key:

```
manifests/<image>/<tag>/
  manifest.json
  history.json
  signatures/
    <key-id>.json        ← deterministic name derived from key fingerprint
```

Multiple signature files are allowed (multiple signers; re-signing after key rotation
keeps both old and new signatures during the rotation window).

### Signature JSON schema

```json
{
  "schemaVersion": 1,
  "digest":    "sha256:abc123...",
  "keyRef":    "awskms://alias/release-signer",
  "keyID":     "arn:aws:kms:us-east-1:123456789:key/mrk-...",
  "algorithm": "ECDSA_SHA_256",
  "signature": "<base64-encoded DER signature>",
  "payload":   "<base64 of signed blob (digest + metadata)>",
  "signedAt":  "2026-04-12T15:00:00Z"
}
```

The signed payload blob is `sha256:<digest>\n` — minimal and unambiguous.

## Library

Use **`github.com/sigstore/cosign/v2/pkg/signature`** which provides a unified
`Signer` / `Verifier` interface across local keys, AWS KMS, GCP KMS, Vault, and PKCS11.

We use only the signing primitives — **not** cosign's OCI registry upload flow.
Signatures are stored in S3 ourselves, consistent with s3lo's design.

Key loading: `signature.LoadSignerVerifier(keyRef, hash.SHA256)` where `keyRef` is
the `--key` flag value. This handles scheme dispatch automatically.

## CLI

### `s3lo sign`

```
s3lo sign <ref> --key <key-ref>

Flags:
  --key string      Signing key: file path, awskms://..., or hashivault://...
  --platform string Select a specific platform for multi-arch images (default: all)
  --output string   Output format: text|json (default: text)

Exit codes:
  0  Signed successfully
  1  Error (key not found, no access, etc.)
```

Text output:
```
Signed myapp:v1.0
  Digest:   sha256:abc123...
  Key:      awskms://alias/release-signer
  Key ID:   arn:aws:kms:us-east-1:123456:key/mrk-...
  Signed:   2026-04-12T15:00:00Z
  Stored:   s3://bucket/manifests/myapp/v1.0/signatures/mrk-....json
```

### `s3lo verify`

```
s3lo verify <ref> --key <key-ref>

Flags:
  --key string      Verification key: file path, public key (.pub), awskms://..., hashivault://...
  --output string   Output format: text|json (default: text)

Exit codes:
  0  Signature valid
  1  Signature invalid, tampered, or missing
  2  Error (cannot reach key backend, etc.)
```

Text output (success):
```
✓ Verified myapp:v1.0
  Digest:   sha256:abc123...
  Signed:   2026-04-12T15:00:00Z
  Key:      awskms://alias/release-signer
```

Text output (failure):
```
✗ Verification FAILED for myapp:v1.0
  No signature found for key awskms://alias/release-signer
```

JSON output on verify (for CI pipelines):
```json
{
  "verified": true,
  "digest": "sha256:abc123...",
  "signedAt": "2026-04-12T15:00:00Z",
  "keyRef": "awskms://alias/release-signer",
  "keyID": "arn:aws:kms:us-east-1:..."
}
```

## Architecture

### New files

| File | Purpose |
|---|---|
| `pkg/image/sign.go` | `Sign(ctx, ref, keyRef string) (*SignResult, error)` |
| `pkg/image/verify.go` | `Verify(ctx, ref, keyRef string) (*VerifyResult, error)` |
| `cmd/s3lo/sign.go` | Cobra command wiring for `sign` |
| `cmd/s3lo/verify.go` | Cobra command wiring for `verify` |

### Flow: `s3lo sign`

1. `Parse(ref)` → resolve bucket, image, tag
2. `NewBackendFromRef(ctx, ref)` → s3 or local backend
3. Read `manifests/<image>/<tag>/manifest.json` → compute digest
4. `signature.LoadSigner(keyRef)` → cosign Signer
5. Sign payload `sha256:<digest>\n`
6. Derive `keyID` from signer (KMS key ARN, or SHA-256 fingerprint of public key)
7. Write `manifests/<image>/<tag>/signatures/<keyid-slug>.json`
8. Return `SignResult{Digest, KeyRef, KeyID, SignedAt, StoredPath}`

### Flow: `s3lo verify`

1. `Parse(ref)` → resolve bucket, image, tag
2. `NewBackendFromRef(ctx, ref)` → backend
3. Read manifest → compute digest
4. `signature.LoadVerifier(keyRef)` → cosign Verifier
5. Derive expected `keyID` slug from verifier
6. Read `manifests/<image>/<tag>/signatures/<keyid-slug>.json`
   - If not found → `VerifyResult{Verified: false, Reason: "no signature found"}`
7. Verify signature bytes against payload using verifier
8. Return `VerifyResult{Verified, Digest, SignedAt, KeyRef, KeyID}`

### `key-id-slug` derivation

- **KMS**: URL-safe last segment of the key ARN (e.g., `mrk-abc123`)
- **Local key**: hex SHA-256 of DER-encoded public key, first 16 chars
- Stored as filename: `<slug>.json`

## `s3lo inspect` integration

`inspect` output gains a `Signatures` field listing all present signature files with their
key references and timestamps. This is read-only; no extra flags needed.

---

## Dependencies

Add to `go.mod`:
```
github.com/sigstore/cosign/v2  v2.x.x
```

Cosign v2 has a large transitive closure. We import only
`github.com/sigstore/cosign/v2/pkg/signature` and
`github.com/sigstore/cosign/v2/pkg/signature/kms` to keep the footprint small.

## Error handling

| Scenario | Behavior |
|---|---|
| Key not found / no IAM access | Clear error message with key reference; non-zero exit |
| Manifest not found | Error "image not found" |
| Signature file missing on verify | VerifyResult{Verified: false}, exit 1 |
| Signature bytes invalid | VerifyResult{Verified: false, Reason: "invalid signature"}, exit 1 |
| KMS / network error | Exit 2 (separate from verification failure) |

## Out of scope

- Keyless / Sigstore transparency log (Rekor)
- Automatic verification on pull (may be added as a flag in v1.10.0+)
- `s3lo list` / `s3lo inspect` badge changes beyond listing present signatures
- Revoking / deleting signatures (delete the file manually or via `s3lo delete`)
